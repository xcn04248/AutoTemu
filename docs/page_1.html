1. API 概述：规模化商品管理的核心

Temu 合作伙伴平台提供的 Product API 是一套专为卖家设计的程序化接口，旨在帮助卖家大规模、高效率地管理其商品目录。通过这套 API，卖家可以实现商品的自动化创建、更新与查询，从而显著提升运营效率。本文将依据 Temu 官方合作伙伴文档，对商品发布流程所涉及的 API 接口、关键字段和注意事项进行全面解析。

核心目标：帮助开发者和卖家理解商品发布的完整技术链路，高效利用 API 完成商品刊登，并规避常见问题。

2. 核心 API 接口矩阵

商品发布并非单一 API 调用，而是一个涉及多个接口协作的流程。这些接口各司其职，共同构成了商品数据的准备、校验和提交闭环。以下是发布流程中涉及的关键 API 接口：

  • 类目与属性获取：
    • bg.local.goods.cats.get: 获取 Temu 平台完整的商品类目树。发布商品时必须使用最末级的“叶子类目”。
    • bg.local.goods.template.get: 根据指定的类目 ID，获取该类目下的商品属性（如颜色、尺寸）和规格（SKU）模板。
  • 媒体资源上传：
    • bg.local.goods.image.upload: 上传商品图片，并返回可用于发布的图片 URI。
    • bg.local.goods.gallery.signature.get: 用于获取上传视频、资质文件等资源的签名，是文件上传的前置步骤。
  • 数据校验与合规检查：
    • temu.local.goods.illegal.vocabulary.check: 检查商品标题、描述等文本信息是否包含违规词汇，强烈建议在提交前使用此接口以避免影响销售。
    • bg.local.goods.out.sn.check: 校验商家自定义的商品编码（out.sn）是否在店铺内重复。
    • bg.local.goods.compliance.property.check: 校验商品属性是否符合站点的销售规则。
  • 核心发布接口：
    • bg.local.goods.add: 创建一个全新的商品。这是整个流程的最终提交动作。

除此之外，还包括获取运费模板、品牌信息、税务代码等辅助接口，共同保障了商品信息的完整性和准确性。

3. 商品发布关键字段详解

在调用 bg.local.goods.add 接口时，请求体中包含了多个复杂的对象和字段。理解这些字段的含义和填写要求至关重要。以下是对核心字段的拆解分析。

3.1 基础信息 (goodsBasic)

此部分为必填项，包含了商品最核心的识别信息。

  • 商品名称 (goodsName): 必填。长度限制在 500 字符以内，仅支持英文字母、数字和部分符号。官方强烈建议使用 temu.local.goods.illegal.vocabulary.check 接口进行预先检查。
  • 类目 ID (catId): 必填。必须通过 bg.local.goods.cats.get 接口递归查询，直至获取到最具体的叶子类目 ID。错误的类目层级会导致发布失败。
  • 外部商品编码 (outGoodsSn): 选填。用于关联卖家自身系统与 Temu 平台的商品，必须在店铺内保持唯一。

3.2 商品媒体资源 (goodsGallery)

此部分为选填项，但高质量的图片和视频是吸引消费者的关键。

  • 详情图 (detailImage): 最多 49 张，宽高均需大于 480px，大小不超过 3MB。图片需先通过 bg.local.goods.image.upload 接口上传转换。
  • 轮播视频 (carouselVideo): 最多 1 个，时长不超过 60 秒，大小不超过 100MB。
  • 详情视频 (detailVideo): 最多 1 个，时长不超过 180 秒，大小不超过 300MB。

注意：所有视频和文件（如资质证明）的上传，都需要先调用 bg.local.goods.gallery.signature.get 获取上传凭证，流程相对复杂。

3.3 服务与物流 (goodsServicePromise)

此部分为必填项，定义了商品的发货时效和方式。

  • 发货时限 (shipmentLimitDay): 必填。指从接到订单到发货所需的天数，通常为 1 或 2 天。
  • 履约类型 (fulfillmentType): 必填。当前固定为“1”，代表“自行履约”。
  • 运费模板 ID (costTemplateId): 必填。需要卖家在商家中心预先创建运费模板，然后通过 bg.freight.template.list.query 接口查询获得。

3.4 商品属性与规格 (goodsProperty)

此部分为必填项，是整个商品发布流程中最为复杂的部分，用于定义商品的销售属性（如颜色、尺寸）和非销售属性（如材质、风格）。

  • 属性来源: 所有属性的结构和可选值都必须通过 bg.local.goods.template.get 接口获取。
  • 属性类型: 需区分 isSale=true 的销售属性（构成 SKU）和 isSale=false 的普通属性。
  • 必填与联动: 必须填写所有 required=true 的属性。同时，要注意属性间的父子联动关系，选择某个父属性值后，可能需要填写其对应的子属性。

3.5 合规与认证信息 (certificationInfo)

此部分在发布时为选填项，但在商品上架销售前必须补充完整，否则销售将受限。这部分信息对于在特定市场（如欧盟、韩国）销售至关重要。

  • 资质要求获取: 需要上传哪些资质（如 CE 认证），取决于商品的类目和属性。必须通过 bg.local.goods.compliance.rules.get 接口查询具体的合规要求。
  • 信息类型: 包括证书文件、检测报告、欧盟能源标签、负责人信息（GPSR Info）等。
  • 文件上传: 证书、报告等文件同样需要通过文件上传接口（涉及 bg.local.goods.gallery.signature.get）提前上传至 Temu。

此外，还有商品亮点 (bulletPoints)、图文描述 (goodsDesc)、尺码表 (goodsSizeChartList) 等多个选填字段，完善这些信息有助于提升商品转化率。

5. 总结与建议

Temu 的 Product API 提供了一套功能强大但结构复杂的商品管理解决方案。对于希望实现自动化上架的卖家和开发者而言，成功对接的关键在于：

  1. 深入理解流程，而非孤立看待接口：商品发布是一个由多个 API 协同完成的链式过程。必须按顺序获取类目、属性模板，再准备数据，最后提交。
  2. 重视数据校验：在调用最终的 bg.local.goods.add 接口前，务必使用 temu.local.goods.illegal.vocabulary.check 等校验接口对内容进行预检查，这能有效减少因数据不合规导致的发布失败和后续风险。
  3. 分步实现，先主后次：优先实现对 goodsBasic、goodsServicePromise 和 goodsProperty 三个必填核心字段的组装与提交，确保主流程畅通。然后，再逐步完善图片、视频、合规信息等选填但重要的补充内容。
  4. 关注合规性：特别是面向特定国家市场销售时，certificationInfo 字段的处理至关重要。应将其视为“准必填项”，在开发时预留出完整的处理逻辑。

通过遵循官方文档指引并结合本文的解析，开发者可以更清晰地规划开发路径，高效、准确地完成与 Temu 平台的商品系统对接，从而在激烈的电商市场中抢占先机。
